RAG Knowledge Chatbot — Architecture
What It Is
A serverless Retrieval-Augmented Generation chatbot with a multi-cloud dual-provider AI architecture (AWS Bedrock + SambaNova). Built with a custom retrieval layer rather than Bedrock Knowledge Bases, giving full control over chunking, embedding, and cosine similarity search.
Tech Stack
  • AWS Lambda, API Gateway, S3, CloudFront, EventBridge
  • AWS Bedrock: Titan Embeddings v2 + Claude Haiku 4.5
  • SambaNova: Llama 3.3-70B Instruct
  • CloudFormation for infrastructure as code
  • Vanilla HTML, CSS, JavaScript frontend — no frameworks
  • Python 3.12 Lambda runtime
Data Layer
  • 2,027-chunk combined vector index stored in S3 as JSON
  • Titan Embeddings v2 (1,536-dimensional vectors)
  • Cosine similarity search runs in-Lambda at query time
  • Two knowledge bases in a single index: Jimmy’s background (resume, project writeups) and AWS Well-Architected Framework (six pillar whitepapers)
  • source_kb field on every chunk enables cross-KB search with visible source attribution
  • Module-level index caching: warm Lambda containers skip S3 reload on subsequent queries
Key Decisions
  • Custom retrieval over Bedrock Knowledge Bases — exposes the embedding, chunking, and search layers instead of abstracting them away
  • Single combined index with source_kb tagging — one cosine pass per query, merge logic is just sorting by score
  • Module-level index caching + 1024MB Lambda memory — cold worst-case dropped from 32s to 15s; EventBridge warmup primes the cache so first-touch is typically ~5s
  • EventBridge warm-up ping every 5 minutes — primes S3 index cache, keeps Lambda alive, synthetic SambaNova call at 3s timeout; ~$0.03/month
  • Bedrock as default provider, SambaNova via toggle — latency is comparable (~2.5–5s), Bedrock produces deeper biographical answers
  • max_tokens=256 matched across both providers — consistent answer length regardless of model selection
  • CloudFormation for every AWS resource — no console clicks, reproducible deployments
Structure
User flow on every query:
  • Frontend sends query text and selected provider to API Gateway
  • Query Lambda embeds the query with Titan Embeddings v2
  • Cosine similarity search against the 2,027-chunk index returns top-K matches with source_kb tags
  • Provider router: Bedrock path uses Claude XML prompt template, SambaNova path uses Llama Markdown template
  • Response returned with generated answer plus retrieved source chunks, relevance scores, and KB badges
  • Right panel renders sources with visible attribution: orange badge for Jimmy’s background, blue badge for Well-Architected
What I Learned
  • Multi-cloud AI provider abstraction — isolating generation from retrieval makes swapping providers trivial
  • Serverless latency engineering — memory tuning, index caching, and EventBridge warm-up each moved the needle independently
  • RAG retrieval quality tuning — chunk size, overlap, and curated knowledge files all affect cosine scores
  • Cost discipline on AWS — under $5/month actual spend on a $20/month budget with observability, CloudFront, and dual AI providers
FieldIQ — Architecture
What It Is
A college football analytics dashboard delivering data-driven stats, standings, coach data, and performance breakdowns. Built as a fully client-side single-page application — no backend, no build step.
Tech Stack
  • Vanilla HTML, CSS, JavaScript — no frameworks
  • Modular JS organized by analytical phase
  • Hosted on AWS S3 + CloudFront
Data Layer
  • CFBD (College Football Data) API — live stats, standings, coach records
  • All data fetched client-side via fetch() at runtime
  • Aggressive caching strategy: past results cached forever, rankings 24hrs, current week 1hr
  • 1,000 req/month free tier — every fetch decision is deliberate
  • Per-section error handling — one failed fetch never crashes the page
Key Decisions
  • Client-side only — avoids backend complexity for a read-only analytics use case
  • API-driven rendering — UI built entirely from live API responses
  • _deriveIdentity() concatenates firstName + lastName from the CFBD /coaches endpoint to normalize coach identity across datasets
Structure
Nine analytical lenses built across 15 phases:
  • Overview: Program DNA labels, Season Snapshot, Last/Next Game, School Identity
  • Game Control: Blowout vs Grinder profile, first/second half scoring, red zone efficiency, turnover margin
  • Volatility: Consistency Rating (std dev), Trap Game Index, largest win/loss
  • Situational: Home/away splits, night game record, bowl history, revenge game flags
  • Market Performance: ATS record (overall/home/away), O/U record, cover streaks
  • Long-Term Strength: Rankings history, NFL draft production, full coaching history from 1995, bowl all-time record
  • Campus Map: Leaflet.js + OpenStreetMap, school-colored pin, click-to-expand modal
  • H2H Search: Any two FBS schools, full record from 1990–present, ATS in matchups
  • Schedule: Full season schedule with spread/O&U, past results with ATS outcome
What I Learned
  • Working with third-party sports APIs — rate limits, inconsistent schemas, missing fields
  • Client-side data transformation at scale without a framework
  • Building modular vanilla JS that stays maintainable across fifteen independent phases
