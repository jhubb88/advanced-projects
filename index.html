RAG Knowledge Chatbot — Architecture
What It Is
A serverless Retrieval-Augmented Generation chatbot that answers questions grounded in real documents. Two knowledge bases live in one combined vector index: a personal/professional KB (resume, project writeups, work history) and the AWS Well-Architected Framework (~2,000 chunks across all six pillars). Users ask plain-English questions; the system retrieves the most relevant chunks via cosine similarity and generates grounded answers using AWS Bedrock.
Tech Stack
  • AWS Lambda (Python) — ingest and query compute
  • AWS Bedrock — Claude Haiku 4.5 for generation, Titan Embeddings v2 for vector embeddings
  • Amazon S3 — document storage, vector index, frontend hosting
  • API Gateway — public HTTPS endpoint
  • CloudFront — global CDN with HTTPS
  • CloudFormation — infrastructure as code
  • CloudWatch — dashboards, alarms, observability
  • SNS — alerting on threshold breaches
  • EventBridge — scheduled Lambda warmup
  • Plain HTML / CSS / Vanilla JS frontend (no framework, no build step)
Data Layer
~2,027 chunks in a single S3-hosted JSON vector index, ~58MB total. Each chunk carries a source_kb tag for cross-KB filtering and UI badging. Documents chunked at 175 words with 20-word overlap, embedded via Titan Embeddings v2 (1,536-dim vectors). Cosine similarity runs in-Lambda — no Bedrock Knowledge Bases, no managed vector DB. Custom retrieval was a deliberate choice to demonstrate the layer rather than abstract it away.
Key Decisions
  • Custom retrieval over Bedrock Knowledge Bases — full control of chunking, ranking, and KB tagging; demonstrates the underlying technique.
  • Cross-KB search with source tagging — single combined index, filter by tag at the UI rather than running per-KB queries.
  • EventBridge warmup ping every 5 minutes — eliminates cold-start latency for demo traffic at ~$0.01/month.
  • Module-level index caching in Lambda — warm containers skip the 58MB S3 load, dropping warm queries to 3–5s.
Structure
Three-panel analyst console, not a chat box. Left: knowledge base navigator. Center: question + answer thread. Right: retrieved source chunks with relevance scores and KB badges. Top bar shows live status. Mobile (iPhone Safari) uses slide-over panels. Designed to look like a tool a real analyst would use, not a portfolio toy.
What I Learned

Originally architected with a secondary external LLM provider to demonstrate cross-cloud routing. The external provider’s free-tier limits and billing system were unreliable enough that the demo would have intermittently failed for recruiters. Made the engineering call to cut the second provider and ship Bedrock-only. The graceful-error patterns built for the dual-provider version (rate-limit detection, fallback UI cards, structured logging of provider responses) transferred directly to single-provider error handling — none of that work was wasted.

Other lessons:

  • Free-tier provider limits should be verified via actual API response headers before architectural decisions, not from marketing pages.
  • iOS Safari has subtle viewport bugs (input auto-zoom under 16px font, 100vh vs 100dvh viewport mismatch) that need handling for mobile-friendly demos.
  • CloudWatch alarm thresholds tuned by raising numbers are toothless; the right lever is evaluation periods (e.g., 3 of 5 datapoints breach) so alarms fire on sustained issues, not blips.
FieldIQ — Architecture
What It Is
A college football analytics dashboard delivering data-driven stats, standings, coach data, and performance breakdowns. Built as a fully client-side single-page application — no backend, no build step.
Tech Stack
  • Vanilla HTML, CSS, JavaScript — no frameworks
  • Modular JS organized by analytical phase
  • Hosted on AWS S3 + CloudFront
Data Layer
  • CFBD (College Football Data) API — live stats, standings, coach records
  • All data fetched client-side via fetch() at runtime
  • Aggressive caching strategy: past results cached forever, rankings 24hrs, current week 1hr
  • 1,000 req/month free tier — every fetch decision is deliberate
  • Per-section error handling — one failed fetch never crashes the page
Key Decisions
  • Client-side only — avoids backend complexity for a read-only analytics use case
  • API-driven rendering — UI built entirely from live API responses
  • _deriveIdentity() concatenates firstName + lastName from the CFBD /coaches endpoint to normalize coach identity across datasets
Structure
Nine analytical lenses built across 15 phases:
  • Overview: Program DNA labels, Season Snapshot, Last/Next Game, School Identity
  • Game Control: Blowout vs Grinder profile, first/second half scoring, red zone efficiency, turnover margin
  • Volatility: Consistency Rating (std dev), Trap Game Index, largest win/loss
  • Situational: Home/away splits, night game record, bowl history, revenge game flags
  • Market Performance: ATS record (overall/home/away), O/U record, cover streaks
  • Long-Term Strength: Rankings history, NFL draft production, full coaching history from 1995, bowl all-time record
  • Campus Map: Leaflet.js + OpenStreetMap, school-colored pin, click-to-expand modal
  • H2H Search: Any two FBS schools, full record from 1990–present, ATS in matchups
  • Schedule: Full season schedule with spread/O&U, past results with ATS outcome
What I Learned
  • Working with third-party sports APIs — rate limits, inconsistent schemas, missing fields
  • Client-side data transformation at scale without a framework
  • Building modular vanilla JS that stays maintainable across fifteen independent phases
Linux Ops Command Copilot — Architecture
What It Is
A Linux-only command copilot for admin and ops workflows. Translates plain-English tasks into Bash, explains them, classifies operational risk, and suggests safer guidance when appropriate. The app never executes commands — it only generates and explains them. Built to demonstrate safety guardrails, structured AI output handling, and AWS serverless architecture as a focused product rather than a generic AI terminal.
Tech Stack
  • React 19 + TypeScript 6 + Vite 8 + Tailwind CSS v4 (CSS-first, `@tailwindcss/vite` plugin)
  • Python 3.12 AWS Lambda — 512 MB, 15 s timeout, Lambda SnapStart with `AutoPublishAlias: live`
  • AWS Bedrock — Claude Haiku 4.5 via `converse()` with native Structured Outputs (`outputConfig.textFormat`, `json_schema`, 14 fields all required)
  • API Gateway HTTP API — throttled at burst 20, rate 10 req/sec
  • AWS SAM + CloudFormation for both backend and frontend stacks
  • S3 + CloudFront frontend — private bucket, Origin Access Control (SigV4), BlockPublicAccess all-true, SPA fallback (403/404 → index.html)
  • CloudWatch logs (7-day retention), $15 billing alarm, SNS alerts, 5-minute EventBridge warmer
Safety Layer

Four-tier risk model — Safe (read-only), Caution (limited state change), Risky (destructive or broad), Refuse (malicious or sabotage intent). Every model response runs through a deterministic post-model safety pass (`backend/src/safety.py`) that ratchets risk classification up but never down.

Six compiled regex families flag dangerous command syntax: destructive_delete, filesystem_disk, permission_ownership, remote_execution, resource_exhaustion, device_writes — with Tier-1/Tier-2 path detection (`/`, `/etc`, `/usr`, `/boot`) and quote stripping so `rm -rf "/"` and `chmod -R 777 '/etc'` still match.

When the model refuses a dangerous-but-legitimate operation, the safety layer enriches the response with a concrete safer alternative, pre-check command, and technical explanation drawn from deterministic templates for 5 task-text families: device_writes, remote_execution, permission_ownership, filesystem_disk, destructive_delete. The pre-model gate (`check_refusal_intent`) and the enrichment gate (`_has_malicious_intent`) use intentionally different keyword lists — the pre-model gate is conservative on overloaded terms like "wipe" / "format" / "delete"; the enrichment gate is narrower (destroy, sabotage, bypass security, fork bomb) so legitimate admin tasks still receive a constructive reframe.

Key Decisions
  • Haiku 4.5 over Sonnet 4.5 — swapped during latency optimization. Haiku is sufficient for the 14-field classification task at ~2x the throughput; Sonnet ARNs retained in IAM for a one-line rollback in `handler.py`.
  • Native Structured Outputs over forced Tool Use — migrated to Bedrock's `outputConfig.textFormat` (`json_schema`) once it went GA. Grammar-based constrained decoding eliminates the Tool Use conversion overhead; schema content byte-for-byte identical (verified via `json.dumps`). ~600 ms best-case improvement.
  • Lambda SnapStart for cold-start reduction — `ApplyOn: PublishedVersions` with `AutoPublishAlias: live`. Restore Duration ~609 ms (includes TLS re-establishment via `@register_after_restore`), Billed Restore Duration 70 ms. Cold start total dropped from ~6.94 s to ~4.3 s.
  • Refuse enrichment in `safety.py`, not in the prompt — the same model that refuses to generate the command would generate inconsistent reframes if asked to fill `safer_alternative` and `recommended_precheck` in the same response. Deterministic Python templates guarantee technically-correct reframes regardless of model behavior variance.
  • Terminal-aesthetic visual direction — the planning doc's "premium navy / ops glass" direction was rejected after Session 4 revealed that floating window chrome fights the terminal metaphor the product is built around. Adopted full-viewport terminal commitment (no chrome, no max-width container, no rounded corners on content).
Structure

Single-pane terminal-style scrollback. Each prompt + response pair persists as a history entry; the active textarea is pinned to the viewport bottom (`h-[100dvh]` to work around the iOS Safari `100vh` viewport bug), past entries scroll above it. Entries are modeled as a discriminated-union `HistoryEntry` (`kind: 'loading' | 'success' | 'error'`). Each entry header is a cyan prompt band (`ops@copilot:~$`) over the task text. The result panel renders all 14 schema fields as labeled `## SECTION` blocks with hyphen bullets, a copy-able command block, and a risk badge with destructive / sudo / read-only modifier dots.

All four risk states render — including enriched Refuse with a populated `safer_alternative`. Mobile (iOS Safari real-device tested) uses `[overflow-wrap:anywhere]` on command blocks, 16 px textarea font (Safari's threshold for not auto-zooming on focus), 44 px tap targets per Apple HIG, and blur-on-submit to dismiss the keyboard once a result lands.

What I Learned
  • Warm latency dropped from 4.5–6 s (Sonnet 4.5) to 2.4–3.6 s by combining three independent levers: a model swap to Haiku 4.5, Lambda SnapStart, and a migration from forced Tool Use to native Structured Outputs. Each lever was deployed and measured separately — none were retroactive guesses.
  • Haiku 4.5 has a model ceiling on safety classification. Three dangerous-but-legitimate sysadmin tasks (dd to /dev/sda, pipe-to-bash, recursive chmod on broad paths) classified as Refuse instead of Risky across two prompt-tuning iterations, including a diagnostic deploy with forced Tool Use that produced the same result — ruling out the API migration as the cause. Iteration was capped at 3 attempts; the deterministic enrichment templates in `safety.py` became the permanent workaround.
  • External code review (Codex) caught a subtle correctness bug in the response handling: an early `isCommandResponse` predicate validated only one field (`risk_level`) while asserting the full 14-field shape — structurally identical to the silent `as` cast it was meant to replace. Resolved by validating every required field, cast-free, via `'key' in obj` narrowing. Type predicates that don't actually predicate are landmines for future refactors.
